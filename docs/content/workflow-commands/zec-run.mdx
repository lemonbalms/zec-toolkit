# /zec run

SPEC 요구사항을 DDD 또는 TDD 방법론으로 구현합니다. Plan-Run-Sync의 두 번째 단계입니다.

import { Callout } from 'nextra/components'

## 사용법

```bash
/zec run SPEC-AUTH-001               # SPEC 구현
/zec run SPEC-AUTH-001 --team        # 팀 모드 (병렬 구현)
```

## 에이전트

| 단계 | 에이전트 | 모델 |
|------|---------|------|
| 분석/계획 | `manager-strategy` | opus |
| DDD 구현 | `manager-ddd` | sonnet |
| TDD 구현 | `manager-tdd` | sonnet |
| 품질 검증 | `manager-quality` | sonnet |
| Git 작업 | `manager-git` | haiku |

## 워크플로우 단계

### Phase 1: Analysis & Planning

`manager-strategy`가 SPEC을 분석하고 실행 전략을 수립합니다. 요구사항 추출, 태스크 식별, 복잡도 추정을 수행합니다.

<Callout>Phase 1에서는 코드 작성이 금지됩니다. 분석과 계획에만 집중합니다.</Callout>

### Phase 1.5~1.8: 구현 준비

- **1.5 Task Decomposition**: 원자적 태스크로 분해
- **1.6 Acceptance Criteria**: 인수 조건을 pending TaskList로 등록
- **1.7 File Scaffolding**: 새 파일 스텁 생성 + LSP 베이스라인
- **1.8 MX Context Scan**: `@MX` 태그 스캔하여 컨텍스트 맵 구축

### Batch Mode 자동 전환

태스크 5개 이상, 파일 변경 10개 이상, 독립 태스크 3개 이상 중 하나라도 충족하면 병렬 구현으로 전환됩니다.

### Phase 2: Implementation

`quality.yaml`의 `development_mode`에 따라:
- **DDD** → `manager-ddd`: ANALYZE-PRESERVE-IMPROVE
- **TDD** → `manager-tdd`: RED-GREEN-REFACTOR

### Phase 2.5: Quality Validation (TRUST 5)

| 상태 | 동작 |
|------|------|
| PASS | Git 작업 진행 |
| WARNING | 리뷰 후 진행 |
| CRITICAL | 구현으로 복귀 |

### Phase 2.9~2.10: 마무리

- MX 태그 업데이트
- Simplify Pass (필수) — 코드 품질 자동 개선

### Phase 3: Git Operations

Conventional Commits + SPEC 참조 + `Fixes #N` 커밋 생성.

### Phase 4: Completion

구현 요약 + 다음 단계: `/zec sync`, 추가 구현, 로컬 검토.
