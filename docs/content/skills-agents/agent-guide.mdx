# 에이전트 가이드

ZEC는 27개 전문 에이전트를 사용하여 작업을 위임합니다. 각 에이전트는 특정 도메인에 특화되어 있습니다.

## 에이전트 선택 의사결정 트리

```
사용자 요청
    │
    ├─ 읽기 전용 탐색? ──────▶ Explore (haiku)
    ├─ 계획/요구사항? ────────▶ spec, strategist, critic (opus)
    ├─ 구현 작업? ────────────▶ executor, deep-executor, backend, frontend (sonnet/opus)
    ├─ 품질/리뷰? ────────────▶ quality, security, tester, verifier (sonnet)
    ├─ 지원 작업? ────────────▶ debugger, build-fixer, docs, git (sonnet/haiku)
    └─ 팀 모드? ──────────────▶ team-reader, team-coder, team-tester (sonnet)
```

## 전체 에이전트 카탈로그

### Planning 레인 (3개, opus)

| 에이전트 | 역할 | 사용 워크플로우 |
|---------|------|---------------|
| `spec` | EARS 형식 요구사항 문서 작성 | `/zec plan` |
| `strategist` | 아키텍처 설계, 기술 평가, 구현 전략 | `/zec run` Phase 1 |
| `critic` | 계획 비판, 리스크 식별, 약점 발견 | 계획 리뷰 |

### Implementation 레인 (5개, sonnet/opus)

| 에이전트 | 모델 | 역할 |
|---------|------|------|
| `executor` | sonnet | 일반 코드 구현, 리팩토링, 기능 추가 |
| `deep-executor` | opus | 복잡한 자율 구현 (다중 모듈, 아키텍처 변경) |
| `backend` | sonnet | API, DB, 서버, 인증 구현 |
| `frontend` | sonnet | UI, 컴포넌트, 상태 관리, CSS |
| `debugger` | sonnet | 근본 원인 분석, 버그 추적, 회귀 격리 |

### Quality 레인 (4개, sonnet)

| 에이전트 | 역할 |
|---------|------|
| `quality` | 코드 품질, 안티패턴, 네이밍, 복잡도, TRUST 5 |
| `security` | OWASP, 취약점, 인증/인가, 시크릿 노출 |
| `tester` | 테스트 전략, 커버리지, 플레이키 테스트 |
| `verifier` | 완료 검증, 증거 수집, 테스트 적절성 |

### Support 레인 (4개)

| 에이전트 | 모델 | 역할 |
|---------|------|------|
| `docs` | sonnet | 문서 작성, 마이그레이션 노트 |
| `git` | haiku | 커밋, 브랜치, PR, 릴리스 관리 |
| `build-fixer` | sonnet | 빌드/타입/린트 에러 수정 |
| `refactorer` | sonnet | 대규모 코드 변환, AST 기반 마이그레이션 |

### Builder 레인 (3개, sonnet)

| 에이전트 | 역할 |
|---------|------|
| `builder-agent` | 에이전트 정의 파일 생성 |
| `builder-skill` | 스킬 정의 파일 생성 |
| `builder-plugin` | 플러그인 생성 |

### Team 레인 (5개, sonnet)

팀 모드(`--team`)에서 사용되는 병렬 협업 에이전트입니다:

| 에이전트 | 역할 |
|---------|------|
| `team-reader` | 코드베이스 분석 (읽기 전용) |
| `team-coder` | 구현 담당 |
| `team-tester` | 테스트 작성 |
| `team-designer` | UI/UX 디자인 |
| `team-validator` | 품질 검증 |

### Utility (1개)

| 에이전트 | 모델 | 역할 |
|---------|------|------|
| `Explore` | haiku | 코드베이스 탐색, 파일/심볼 검색 |

## 모델 라우팅 규칙

| 복잡도 | 모델 | 비용 | 예시 |
|--------|------|------|------|
| Simple | **haiku** | 최저 | Explore, git, 빠른 조회 |
| Standard | **sonnet** | 중간 | 구현, 디버깅, 리뷰, 테스트 |
| Complex | **opus** | 최고 | spec, strategist, deep-executor |

## executor vs deep-executor

| | executor | deep-executor |
|---|---|---|
| 모델 | sonnet | opus |
| 용도 | 명확한 지시 실행 | 자율적 판단 필요 |
| 범위 | 단일 기능, 5파일 이하 | 다중 모듈, 20+ 파일 |
| 예시 | 로그인 폼 유효성 검사 | 인증 시스템 전체 리팩토링 |
